# Janus v0.2 — Harness-Native Inbox Digest

## Thesis

v0.2 is not a new Janus application surface.

It is an agent skill that runs inside an existing coding-agent harness and uses the harness’s native capabilities:

```text
filesystem reads
shell commands
repository search
conversation state
local operational checklist updates
guarded file edits
```

The primary implementation artifact is:

```text
.claude/commands/digest.md
```

The command is the agentic workflow.

Janus provides the repository conventions, boundaries, and verification rules. Claude Code or Codex provides the execution loop.

## Scope

`/digest` processes root inbox notes only.

Its authoritative work queue is:

```sh
pnpm brain:inbox -- --json
```

It must not process, read for digestion, or propose changes to:

```text
journal/**
```

Journal-based reflection and weekly review input are deferred to v0.3.

## Agent roles

The command creates one coordinating agent role.

### Digest coordinator

The coordinator owns:

```text
digest state
user interaction
knowledge-extraction questions
organization decisions
approval gates
ledger updates
final edits
verification
```

Only the coordinator may ask the user questions, modify `.janus/digest-ledger.md`, or modify knowledge-source files after approval.

v0.2 does not use retrieval scout subagents. The coordinator performs retrieval itself.

## Digest state machine

The command is a stateful multi-turn agent loop.

```text
DISCOVER
  -> INVENTORY
  -> TEACH
  -> RETRIEVE
  -> REVISE
  -> ORGANIZE
  -> RESOLVE_AMBIGUITY
  -> PROPOSE_CHANGESET
  -> WAIT_FOR_APPROVAL
  -> APPLY
  -> VERIFY
```

The agent must keep two ledgers:

```text
conversation ledger -> reasoning state
.janus/digest-ledger.md -> local operational checklist
```

The conversation ledger contains:

```text
active inbox notes
current note
teacher status per note
known note nucleus
retrieval candidates
open knowledge gaps
organization disposition
open user decisions
approved changes
```

The local checklist is created or resumed with:

```sh
pnpm brain:digest:init
```

It uses:

```md
# Janus Digest Ledger

- [ ] note-a.md
- [x] note-b.md — clarified
- [x] note-c.md — parked
- [x] note-d.md — keep raw
- [x] note-e.md — proposed discard
```

`[x]` means no further teacher/retrieval work is required for this digest run. It does not mean a final changeset has been approved or applied.

The initializer behavior is:

```text
No ledger exists          -> create it
Ledger exists             -> preserve it and report that it is being resumed
Explicit --reset          -> overwrite it
```

Reset command:

```sh
pnpm brain:digest:init -- --reset
```

At startup, `/digest` must compare current `pnpm brain:inbox -- --json` paths against ledger paths:

```text
Exact match       -> resume
Mismatch          -> explain added/removed paths and ask whether to reset
```

The agent must never silently reconcile a stale ledger or discard prior checklist state.

At the beginning of each user-facing turn, show only this compact progress line:

```text
Digest: 2/5 notes clarified · 1 parked · 2 open organization decisions
```

## Teacher loop

The teacher loop is not ordinary clarification.

Its purpose is to actively extract knowledge that exists in the user’s head but was omitted from the raw capture.

The teacher loop is mandatory for every unresolved inbox note. The command must not skip directly from inventory to organization.

At the start of this phase, the agent should print:

```text
Phase 2 — Teaching: <path>
```

For each active note, the agent must:

1. Read the current source note in full.
2. Classify its provisional shape.
3. State a concise interpretation.
4. Search relevant durable notes using harness-native repository search tools.
5. Inspect the full contents of plausible candidate notes.
6. Identify the highest-value missing knowledge.
7. Show a compact teaching card:

```text
Teaching note: <path>
Provisional shape: <shape>
Current interpretation: <1-3 bullets>
Retrieved context: <paths or "none found">
Highest-value gap: <specific missing knowledge>
```

8. Use the harness `ask` tool to ask exactly one targeted question.
9. Stop and wait for the user’s answer.
10. Update the note’s working understanding after the answer.
11. Re-run retrieval if the answer introduces new concepts, constraints, or references.
12. Continue only until the agent can propose a safe disposition and explain its reasoning.

For trivial captures such as a bare link, task, or reminder, the teacher loop still happens. The question should be only the minimum needed to decide whether the note should remain raw, become durable knowledge, or be discarded.

Possible provisional shapes:

```text
idea
claim
decision
observation
reference
project context
task or follow-up
question
fragment
```

### Teacher question selection

The agent must choose questions based on an explicit knowledge gap.

It should prefer questions that force reconstruction of:

```text
intent
context
mechanism
evidence
constraints
counterexamples
consequences
application
relationship to existing knowledge
```

Do not ask generic prompts such as:

```text
Can you explain more?
What do you mean?
Could you provide more context?
```

Instead ask a concrete question grounded in the note and retrieval evidence.

The command must use the harness `ask` tool for teacher questions instead of asking as plain chat text. The `ask` prompt should name the concrete note, the evidence or ambiguity that creates the question, and concise options when useful.

Examples:

```text
You wrote that “agent memory should remain Markdown-first.” What failure mode are you trying to avoid: lock-in, retrieval opacity, editing friction, or something else?

This note says “do not use RAG yet.” Is that a temporary sequencing decision, or a principle that applies even once Janus has more notes?

You mention that root captures may be wrong. What should happen when a digest discovers that the captured conclusion conflicts with a durable note?

This overlaps with brain/projects/janus/architecture.md. What information would this new note add that the architecture document should not own?
```

### Teacher completion threshold

A note is ready for organization once the agent can propose a safe disposition and explain its reasoning without guessing.

The agent should prefer:

```text
safe disposition
  > sufficient understanding
  > complete understanding
```

The agent should know enough to state:

```text
what the note means
why it matters
whether it is a fact, hypothesis, decision, question, or reference
what conditions or evidence support it
how it relates to retrieved Janus knowledge
what durable outcome or inbox outcome is being proposed
```

Not every field must be known.

Unknown non-critical context should be recorded as uncertainty, not converted into more questions.

Do not continue questioning merely to complete a conceptual template.

### User controls

The user may reply with:

```text
skip
park
keep raw
discard
stop
```

Interpretation:

```text
skip      -> record the unanswered gap and move to the next best question
park      -> stop processing this note; leave it unchanged in root inbox
keep raw  -> treat the note as intentionally unrefined; leave it in root inbox
discard   -> mark it as a proposed discard; require final confirmation before deletion
stop      -> end the digest without changing files and show the current ledger
```

## Retrieval protocol

v0.2 retrieval is agentic repository inspection, not a search feature.

For each non-trivial note, the coordinator must:

1. Extract likely concepts, proper nouns, project names, and decision terms.
2. Search `brain/**/*.md` using harness-native search tools.
3. Inspect the full contents of plausible candidate notes.
4. Compare claims, scope, and ownership boundaries.
5. Return a small evidence set.

The coordinator must not use shell `grep`, `find`, `ls`, or pipelines for digest retrieval or inventory. Use harness-native search, file listing, and read tools.

Each retrieval result must contain:

```text
path
location class
relevant excerpt
relationship
confidence
```

Allowed relationships:

```text
supports
refines
contradicts
duplicates
contains
historical_context
weakly_related
```

The agent must not claim duplication based only on a title or keyword match.

It must inspect the full candidate before proposing a merge or overwrite.

`brain/archive/**` is historical context. It is not the default target for edits.

## Organization loop

Do not organize a note while the teacher loop is still incomplete.

After all active notes are clarified, the agent creates an organization proposal for each note.

Use this structure:

```text
Source note:
Extracted knowledge:
Evidence status:
Relevant Janus notes:
Relationship:
Recommended disposition:
Recommended source handling:
Reasoning:
Proposed destination:
Proposed title:
Open decision:
```

Allowed dispositions:

```text
update_existing_note
merge_into_existing_note
create_new_durable_note
keep_in_inbox
park_for_later
discard_with_confirmation
```

Allowed source handling:

```text
preserve_in_root
archive_after_promotion
delete_with_confirmation
```

Approved archived root captures go under:

```text
brain/archive/inbox/
```

If the archive destination already exists, the agent must stop and ask the user for a destination. It must not overwrite or auto-merge archived captures.

If an inbox note conflicts with durable `brain/` knowledge and the user does not resolve the conflict, default to `park_for_later` and leave the source note in root inbox.

Default preference:

```text
update or merge before creating a new durable note
```

The agent must not invent a new tag, taxonomy, folder, or note category.

When note placement is materially ambiguous, ask the user one organization question and stop.

Examples:

```text
This could extend brain/projects/janus/architecture.md or become a new note on agent workflow design. Which should own the concept: Janus architecture or a reusable workflow note?

The note is related to two existing notes but does not clearly belong to either. Should we preserve it as an independent concept, or merge it into the architecture note as an implementation constraint?
```

## Approval and write protocol

The command is read-only for knowledge-source files until the user explicitly approves a complete proposed changeset.

Before approval, the agent may modify only:

```text
.janus/digest-ledger.md
```

Before approval, the agent must not modify:

```text
inbox notes
journal notes
brain notes
archive notes
templates
source code
project documentation
```

Before writing knowledge-source files, present:

```text
Create:
Update:
Move or archive:
Keep in inbox:
Discard:
```

The agent may apply knowledge-source changes only after the user gives explicit approval for that changeset.

After approval, the agent must:

1. Re-read every target file before editing.
2. Apply only approved changes.
3. Preserve the original root capture unless the approved plan explicitly moves, archives, or deletes it.
4. Check archive destination collisions before moving captures into `brain/archive/inbox/`.
5. Never overwrite unrelated content.
6. Run:

```sh
pnpm brain:index
```

7. Inspect the resulting diff.
8. Report every changed file and any uncompleted proposal.

No write may be justified merely because the agent considers it obvious.

## Command file

Save the shipped slash command as:

```text
.claude/commands/digest.md
```

The command file must implement this sketch's workflow and include:

```text
description: Interactively digest Janus root inbox notes into clarified, non-duplicated durable knowledge. Use for `/digest`.
```

Required command behavior:

1. Act as the Janus digest coordinator.
2. Process only notes returned by `pnpm brain:inbox -- --json`.
3. Run `pnpm brain:digest:init` at startup to create or resume `.janus/digest-ledger.md`.
4. Run `pnpm brain:digest:init -- --reset` only after explicit user instruction.
5. Compare ledger paths against current inbox paths before resuming.
6. Use the harness `ask` tool before resetting or reconciling a stale ledger.
7. Do not process, read for digestion, summarize, or modify `journal/**`.
8. Do not use retrieval scout subagents in v0.2.
9. Maintain detailed reasoning state in conversation.
10. Use `.janus/digest-ledger.md` only as a file checklist checkpoint.
11. Before moving to another inbox note, update the current note's checklist line to one of:

```md
- [x] path.md — clarified
- [x] path.md — parked
- [x] path.md — keep raw
- [x] path.md — proposed discard
```

12. Before approval, modify only `.janus/digest-ledger.md`.
13. Before approval, do not modify inbox notes, journal notes, brain notes, archive notes, templates, source code, or project documentation.
14. Use the harness `ask` tool for each targeted teacher question, organization ambiguity, archive collision, and final approval.
15. Do not claim duplication, contradiction, or mergeability until the full candidate note has been read.
16. Present a complete changeset before any knowledge-source edits.
17. After approval, re-read targets, apply only approved edits, check archive destination collisions, run `pnpm brain:index`, inspect the diff, and report changed files.

## Verification cases

v0.2 is complete only after manually exercising the command against fixture notes or documented sample notes covering:

```text
a raw idea with missing context
a note that clearly refines an existing durable note
a note with two plausible durable homes
a bare reference or link
a note that should remain in inbox
an inbox note that conflicts with existing knowledge
a journal note that must not appear in the digest queue
a stale digest ledger whose paths differ from the current inbox queue
an archive destination collision
```

Main acceptance criterion:

```text
For each inbox note, the agent reads enough evidence and asks only enough targeted questions to propose a safe disposition. It does not require complete knowledge before progressing.
```

For each case, verify:

1. `/digest` uses `brain:inbox` as its queue.
2. `brain:digest:init` creates or resumes the inbox checklist.
3. The agent inventories queued notes before the first interaction without reading every queued note in full.
4. The agent reads the current full note before asking about it.
5. The agent performs repository retrieval before proposing a destination.
6. Teacher questions are specific, information-seeking, and limited to what is needed for a safe disposition.
7. The agent asks before resolving genuine organizational ambiguity.
8. The agent updates `.janus/digest-ledger.md` before moving to another note.
9. No knowledge-source file changes occur before explicit approval.
10. Archive destination collisions stop for user input instead of overwriting.
11. The post-approval diff matches only the approved changes.
