# Janus v0.3 Sketch — Daily Planning Loop

## Status

Draft for the next Janus slice after the root-note digest workflow.

## Goal

Turn the daily journal into a deliberate operating loop:

```text
capture task → backlog queue
backlog + prior-day context + calendar → /checkin
today’s active work + notes → /checkout
checkout → reflection, handoff, task reconciliation, daily-note digest
```

v0.3 adds a narrow Markdown-native task workflow. It must not become a general task-management product.

## Existing model preserved

```text
journal/      = chronological working memory
root Markdown = standalone rough inbox captures
brain/        = promoted durable knowledge
data/         = disposable generated data
```

Journal filenames remain the authoritative journal identity. Root captures remain eligible for the existing digest workflow. `brain/` remains deliberate durable knowledge.

## Locked decisions

1. Implement v0.3 as one staged plan, not as one giant cutover.
2. Add root `backlog.md` as the canonical queue of unresolved, uncommitted tasks.
3. Add three harness-native workflows:
   - `/add-task`
   - `/checkin`
   - `/checkout`

4. Add deterministic task CLIs under `pnpm brain:task:*`; slash workflows orchestrate and call CLIs after user approval.
5. Use stable, automatically allocated sequential task IDs: `J-001`, `J-002`, and so on.
6. Keep one flat backlog; do not require `Ready`, `Later`, or `Waiting` categories.
7. Preserve the initial context, constraints, and rough thinking directly beneath each backlog task.
8. Use `estimate: quick|medium|large` as the parseable estimate format.
9. A task has exactly one **active task block**:
   - in `backlog.md`; or
   - in one journal’s `## Todo`.

10. `/add-task` always writes new tasks to `backlog.md`.
11. `/checkin` is the agent-led morning planning flow. It creates today’s journal when missing, reconciles unfinished tasks from the latest prior journal, recommends the day, asks for approval, and applies one structured plan.
12. `/checkout` marks completed tasks, leaves unfinished tasks in the journal, writes end-of-day reflection, and performs a selective daily-note digest.
13. Unfinished tasks are not mechanically rolled into tomorrow. The next `/checkin` must explicitly recommit, return, split, cancel, or complete them.
14. Every `/checkout` performs a selective digest of the current daily note.
15. Add a read-only local Google Calendar `.ics` integration as an optional planning input. `/checkin` must still work when calendar input is unavailable.
16. Delete the unused `pnpm brain:checkin` wellbeing report command and remove current wellbeing reporting tooling.
17. Do not add task state, calendar contents, journal todos, reflections, or digest results to `data/brain-index.json`.

## Core task invariant

```text
An active task is the complete top-level Markdown task block.

Each open task ID exists in exactly one active location:
- backlog.md; or
- one journal/YYYY-MM-DD.md → ## Todo.

Never both.
```

Task ID references inside `## Check-in`, `## Checkout`, or historical notes are not active task copies. They are planning or audit references only.

## Backlog

`backlog.md` is a reserved root system file. It is excluded from inbox hygiene and classified as a protected root document.

```md
# Backlog

<!-- janus-backlog: next_task_id=7 -->

- [ ] [J-001] Finish Janus v0.3 daily-workflow design
  - added: 2026-06-30
  - estimate: medium
  - deadline:
  - blocked_by:
  - context:
    - Need settle the relationship between backlog tasks, daily todos,
      check-in planning, checkout reflection, and daily-note digestion.
    - Local ICS export is sufficient for v0.3; defer OAuth.
  - updates:
    - 2026-07-01: Calendar recurrence handling is necessary for useful
      daily capacity estimates.

- [ ] [J-002] Define local ICS calendar integration
  - added: 2026-06-30
  - estimate: medium
  - deadline:
  - blocked_by:
  - context:
    - Read-only Google Calendar export.
    - Must handle time zones, recurring events, and all-day events.
```

### Backlog grammar

A task block is:

```text
top-level Markdown checkbox line containing [J-###]
+ every indented line that follows
+ until the next top-level task block or end of file
```

Example:

```md
- [ ] [J-014] Research weekly review workflow
  - added: 2026-07-03
  - estimate: large
  - context:
    - Need a weekly ritual that reviews unfinished tasks,
      journal extraction, and project direction.
  - references:
    - journal/2026-07-03.md
```

The parser preserves unknown indented Markdown. `backlog.md` remains user-owned Markdown rather than a rigid internal database.

### Stable task IDs

The exact HTML comment is the ID-allocation source of truth:

```md
<!-- janus-backlog: next_task_id=7 -->
```

Rules:

- `/add-task` uses the current number, creates `J-007`, then increments the counter to `8`.
- IDs are zero-padded for display.
- IDs are never reused, including after completion, cancellation, or deletion from the active backlog.
- The counter must be present and valid once v0.3 initializes `backlog.md`.
- Task IDs remain stable when title, context, effort, or blocker details change.

### Optional task metadata

The task title is mandatory. These fields are optional:

```text
added
estimate
deadline
blocked_by
context
updates
references
context_note
```

Task creation must remain cheap. These fields are prompts, not required classification work.

## Task location and lifecycle

```text
backlog.md
= unresolved tasks waiting for a future commitment

journal/YYYY-MM-DD.md → ## Todo
= tasks currently committed to this specific day
```

### `/add-task`

```text
new task
  → append task block to backlog.md
```

### `/checkin`

```text
selected task
  → remove its full task block from backlog.md
  → insert that same task block under today’s ## Todo
  → write day-specific planning separately under ## Check-in
```

Example before check-in:

```md
# Backlog

<!-- janus-backlog: next_task_id=3 -->

- [ ] [J-001] Finish Janus v0.3 daily-workflow design
  - estimate: medium
  - context:
    - Need settle backlog, daily Todo, check-in, checkout, and digest semantics.

- [ ] [J-002] Define local ICS calendar integration
  - estimate: medium
```

Example after selecting `J-001`:

```md
# Backlog

<!-- janus-backlog: next_task_id=3 -->

- [ ] [J-002] Define local ICS calendar integration
  - estimate: medium
```

```md
# 2026-07-01

## Todo

- [ ] [J-001] Finish Janus v0.3 daily-workflow design
  - estimate: medium
  - context:
    - Need settle backlog, daily Todo, check-in, checkout, and digest semantics.
```

### `/checkout`

`/checkout` closes the day; it does not return unfinished task blocks to `backlog.md`.

For every active task block in today’s `## Todo`, `/checkout` only needs to determine whether it was completed.

| Outcome | Journal result | Backlog result |
| --- | --- | --- |
| Completed | Keep full task block; mark `[x]` | Do not return it |
| Not completed | Leave full task block unchecked | No checkout-time change |

Unfinished task blocks remain in the journal overnight. The next `/checkin` reconciles them before selecting new backlog work.

Completed tasks remain in the journal as historical evidence:

```md
## Todo

- [x] [J-001] Finish Janus v0.3 daily-workflow design
  - estimate: medium
  - context:
    - Need settle backlog, daily Todo, check-in, checkout, and digest semantics.
```

## Carryover at next `/checkin`

The next `/checkin` scans the latest prior journal containing unfinished active task blocks.

Carryover flow:

```text
1. Find the latest earlier journal containing unresolved active Todo task blocks.
2. Ask for a disposition for each carried-over task:
   - recommit today;
   - return to backlog;
   - complete;
   - split;
   - cancel.
3. Append a factual carryover decision under that prior journal’s ## Checkout.
4. Move, recreate, or close task blocks according to the approved check-in plan.
5. Continue normal planning with the remaining backlog.
```

Example carryover audit in the prior journal:

```md
## Checkout

- task_decisions:
  - [J-001] carried forward by /checkin on 2026-07-02 and recommitted to that day
```

For task splitting, each child task receives a fresh ID and an explicit destination:

```md
## Checkout

- task_decisions:
  - [J-004] split by /checkin on 2026-07-02 into [J-005] and [J-006]
    - [J-005] recommitted to journal/2026-07-02.md
    - [J-006] returned to backlog.md
```

This preserves the single-active-location invariant without pretending that unfinished work was completed during checkout.

## `/add-task`

`/add-task` captures future work while preserving the context that task titles normally lose.

### Required interaction

```text
1. What is the task?
2. Is there a deadline, trigger, or real time constraint?
3. Roughly how large is it?
   - quick: <= 30m
   - medium: 30m–2h
   - large: > 2h or exploratory
4. Is it blocked by a person, dependency, or event?
5. Do you want to save notes, initial ideas, a rough breakdown, links, or constraints?
```

Only the title is required.

### Context prompt

When context is wanted, ask:

```text
What do you already have in mind?

- Why does this matter?
- What approach are you considering?
- What should future-you remember?
- What is the first thing worth checking?
- Are there names, files, links, risks, or constraints?
```

Store the response as rough Markdown under the task’s `context` field. Do not force polished prose or false certainty.

### Adaptive capture

Simple task:

```text
Buy cat food
```

Keep the interaction short: title, optional deadline, optional note.

Large or vague task:

```text
Design the Janus v0.3 workflow
Research local calendar integration
Learn distributed systems
```

Offer:

```text
This sounds larger than one sitting. Do you want to save a rough breakdown now, or leave it for /checkin when you select the task?
```

`/add-task` does not decide whether a task is ready, important, or appropriate for today.

## Journal template

New journal notes use this structure:

```md
---
title: { { date:YYYY-MM-DD } }
created: { { date:YYYY-MM-DD } }
kind: journal
---

# {{date:YYYY-MM-DD}}

## Check-in

- capacity:
- primary_outcome:
- calendar_summary:
- constraints:

## Todo

- [ ]

## Notes

## Checkout

- wellbeing:
- worked:
- improve:
- handoff:
- next_step:
- task_decisions:
- digest:
```

## `## Check-in`

`## Check-in` contains morning context and task planning. It is not a duplicate active task list.

| Field              | Meaning                                               |
| ------------------ | ----------------------------------------------------- |
| `capacity`         | Low, normal, or high realistic energy/focus capacity  |
| `primary_outcome`  | The single outcome that would make the day successful |
| `calendar_summary` | Commitments and realistic usable focus time           |
| `constraints`      | Limits not represented in the calendar                |

For selected substantial tasks, `/checkin` writes a day-specific plan:

```md
## Check-in

- capacity: normal
- primary_outcome: Settle the Janus v0.3 workflow.
- calendar_summary: ~3 hours of usable focus time after meetings.
- constraints: Need a reviewable draft before the end of the day.

### Plan: [J-001] Finish Janus v0.3 daily-workflow design

- done_for_today: A reviewable workflow sketch exists.
- first_step: Define the backlog and daily Todo relationship.
- steps:
  1. Define the task-movement lifecycle.
  2. Define check-in planning.
  3. Define checkout and digest outcomes.
- risks:
  - Calendar integration may expand scope.
- stopping_point: The workflow is settled even if implementation details remain open.
```

The plan remains in the journal even if the active task remains unfinished and is reconciled by a later `/checkin`. It is historical planning context, not a second task block.

## `## Checkout`

`## Checkout` is an end-of-day closure record.

| Field            | Prompt                                                            |
| ---------------- | ----------------------------------------------------------------- |
| `wellbeing`      | What was today’s wellbeing score from 1–5?                        |
| `worked`         | What helped or went well?                                         |
| `improve`        | What is one process adjustment worth making?                      |
| `handoff`        | Where did work actually stop?                                     |
| `next_step`      | What is the smallest real restart action tomorrow?                |
| `task_decisions` | How did active tasks resolve?                                     |
| `digest`         | What was extracted, updated, or deliberately kept in the journal? |

Example:

```md
## Checkout

- wellbeing: 3
- worked: I settled the single-location task model.
- improve: Start the main design task before handling small admin work.
- handoff: The remaining question is the calendar CLI contract.
- next_step: Define JSON output for one planning day.
- task_decisions:
  - [J-001] completed
    - result: settled the single-location task model
- digest:
  - backlog: updated [J-002] with recurrence and time-zone requirements
  - inbox: created calendar-integration-questions.md
  - journal_only: kept rough reflection notes in this journal
```

## Wellbeing

v0.3 records wellbeing under exact `## Checkout`, where it belongs as an end-of-day reflection.

`pnpm brain:checkin` is removed. Janus no longer ships a wellbeing reporting CLI in v0.3; `/checkout` records the score as journal content.

## `/checkin`

### Inputs

`/checkin` reads:

1. Today’s journal, creating it from `templates/journal.md` when missing.
2. Unfinished active tasks from the latest prior journal.
3. The prior journal’s `## Checkout`, especially `handoff` and `next_step`.
4. Remaining tasks in `backlog.md`.
5. Calendar availability from `pnpm brain:calendar`, when configured.
6. User-stated constraints not represented in Janus or the calendar.

### Dialogue

`/checkin` is agent-led. The user invokes it before starting work; the agent reconstructs morning context and proposes the day.

The agent asks only for missing judgment:

```text
1. Capacity or constraints not visible in Janus or the calendar.
2. The primary outcome if it cannot be inferred safely.
3. Approval or correction of the proposed plan.
```

After approval, `/checkin` previews one structured apply plan with `--dry-run --json`, then applies the same plan.

### Recommendation rules

The workflow makes transparent recommendations rather than calculating a hidden priority score.

Selection order:

1. Hard deadlines and time constraints.
2. Carried-over work with an explicit prior-day handoff or next step.
3. Work that directly supports today’s primary outcome.
4. Work whose `quick|medium|large` estimate fits available focus.
5. One smaller maintenance or personal task when capacity permits.
6. Stale work only after confirming it is still worthwhile.

Example:

```text
Today has roughly 3 hours of usable focus time.

Recommended plan:
1. [J-001] Finish Janus v0.3 daily-workflow design
   Reason: it matches the primary outcome and has a concrete handoff.

2. [J-002] Define local ICS calendar integration
   Reason: it is a bounded follow-up that fits the remaining focus block.

Not selected:
- [J-003] Research weekly review workflow
  Reason: it is exploratory and does not fit today’s capacity.
```

### Carryover dispositions

For unfinished tasks from the latest prior journal, `/checkin` may:

```text
recommit today
return to backlog
mark completed
split
cancel
```

Split tasks receive fresh IDs. Each child task may be placed either in today’s `## Todo` or `backlog.md`.

### Task-breakdown Q&A

For selected work that is large, vague, exploratory, blocked, or not yet executable, `/checkin` asks:

```text
1. What does “done enough for today” look like?
2. What is the first concrete action?
3. What are the likely steps or important unknowns?
4. What could block progress?
5. What is the stopping point for today?
```

The command should help based on task clarity:

```text
Known task:
- Refine the user’s proposed next step.

Partly known task:
- Propose a rough sequence; let the user correct it.

Unknown or exploratory task:
- Define a discovery step rather than inventing a fake full plan.
```

Only selected substantial tasks receive this dialogue. `/checkin` must not become a planning session for the entire backlog.

### Output

`/checkin`:

1. Creates today’s journal when missing.
2. Resolves unfinished tasks from the latest prior journal.
3. Moves selected backlog task blocks into today’s `## Todo`.
4. Writes morning context under `## Check-in`.
5. Adds task plans where useful.
6. Leaves non-selected tasks in `backlog.md`.

## Calendar integration

v0.3 defines a local, read-only Google Calendar export path:

```text
.janus/
└── calendar/
    ├── primary.ics
    └── config.json
```

`.janus/calendar/` is ignored by Git.

### Calendar CLI

```text
pnpm brain:calendar -- --date YYYY-MM-DD
pnpm brain:calendar -- --date YYYY-MM-DD --json
```

Minimum behavior:

- timed events;
- all-day events;
- recurring occurrences;
- recurrence exceptions;
- calendar time-zone handling;
- merged busy intervals;
- free blocks inside configured planning hours;
- a clear unavailable state when no local calendar export exists.

Initial default planning window:

```text
09:00–22:00 in configured local calendar time
```

Calendar data is advisory. `/checkin` must work without it and ask for realistic capacity instead of inventing availability.

Out of scope:

- Google OAuth;
- live calendar sync;
- calendar writes;
- invitations;
- reminders;
- notifications;
- multi-calendar management.

## `/checkout`

### Step 1: mark completed tasks

For every active task block in today’s `## Todo`, `/checkout` determines whether it was completed:

```text
completed
not completed
```

Completed tasks are marked `[x]` and remain as full task blocks in the journal. Unfinished tasks remain `[ ]` in the journal for the next `/checkin` to reconcile.

`/checkout` does not move unfinished tasks to `backlog.md`.

### Step 2: fixed reflection prompts

```text
1. What was your wellbeing score today, from 1 to 5?
2. What helped or worked well?
3. What created friction or could improve?
4. What should tomorrow-you know before starting?
5. What is the smallest real next step?
```

### Step 3: daily-note digest

Every checkout performs a selective digest of the current journal note.

It examines:

- `## Notes`;
- task outcomes;
- decisions;
- references and links;
- unresolved ideas;
- checkout handoff and next step.

Each meaningful item receives one disposition:

| Finding                                   | Action                                                      |
| ----------------------------------------- | ----------------------------------------------------------- |
| Open work that survives today             | Add or update `backlog.md` through task CLIs                |
| Rough standalone idea, link, or reference | Create a root inbox capture                                 |
| Clear correction to a known durable note  | Propose a `brain/` edit and require explicit approval       |
| Ambiguous or purely temporal context      | Keep in the journal only                                    |

The rule is:

```text
Every daily note is reviewed.
Not every daily note creates an inbox capture or durable note.
```

New root captures enter the existing root-note digest workflow.

When extracting journal content, preserve a source trail:

```md
Source: journal/2026-07-01.md — Notes
```

Durable `brain/` edits are approval-gated. Root inbox and durable knowledge edits use guarded harness edits after approval; task completion and checkout reflection use deterministic task CLIs.

### Step 4: write closure

`/checkout` updates `## Checkout` with:

- wellbeing;
- reflection;
- handoff;
- next step;
- completed task decisions;
- digest audit trail.

## Command editing model

The workflows may edit the appropriate Markdown files during normal operation.

They preserve these correctness invariants:

```text
1. Historical journal records remain truthful.
2. Each open task has one active task block.
3. Unresolved work is never silently discarded.
4. Extracted knowledge retains a source trail to the originating journal note.
```

## Non-goals

Do not implement:

- automatic next-day task rollover;
- automatic task scheduling;
- task priorities or opaque priority scores;
- reminders or notifications;
- Kanban boards;
- generic task dependencies;
- Google OAuth or live sync;
- calendar writes;
- automatic promotion of every daily-note fragment into `brain/`;
- semantic search, embeddings, graph features, or databases.
