# Janus v0.1 Implementation Plan

## Status

Planning document for implementing [`v0.1.md`](./v0.1.md).

## Locked decisions

1. Implement the full v0.1 scope now: journal conventions, documentation, journal discovery, index schema v2, wellbeing check-in reporting, and tests.
2. Use a hard index cutover to `schema_version: 2`; do not preserve a v1 emission or compatibility path.
3. Add a public `--to YYYY-MM-DD` option to `brain:checkin` so reports and tests can anchor the ending date deterministically.
4. Keep `brain:checkin` wellbeing-only in v0.1; do not add a public `--metric` option yet.
5. Treat the journal path as authoritative. Frontmatter in journal notes is template metadata, not a second source of truth.
6. Parse check-in entries with the explicit grammar in Phase 5 while keeping metric keys case-sensitive.
7. In human mode, print the check-in report to stdout and non-fatal warnings to stderr, exiting `0`.
8. Track the empty `journal/` directory with `journal/.gitkeep`; do not place workflow documentation inside `journal/`.
9. Use date-only civil calendar arithmetic for check-in windows; do not use JavaScript timestamp semantics for `YYYY-MM-DD` window math.

## Pushback applied to the sketch

The sketch is intentionally broad. Because v0.1 is being implemented as one complete cut, the implementation should be phased and verified at each layer rather than shipped as a loose batch edit.

Specific constraints to preserve:

- Do not make journal frontmatter authoritative. The filename date in `journal/YYYY-MM-DD.md` is the canonical journal date.
- Do not expose generic metric reporting before a second real metric exists.
- Do not hide date anchoring behind test-only behavior; `--to` is part of the CLI contract.
- Do not turn Janus into a task manager: no rollover, global task aggregation, priorities, due dates, reminders, or task status systems.
- Do not place docs or examples inside `journal/`; it should contain user daily notes plus `.gitkeep` only.
- Do not persist parsed check-in values, todos, notes, averages, or task history in `data/brain-index.json`.
- Do not store journal filename warnings in `frontmatter_warnings`; schema v2 should use a neutral `warnings: DocumentWarning[]` field for document-level warnings.
- Do not let invalid journal filenames create unrelated noise in every check-in report; `brain:checkin` reports metric-content warnings only from valid dated journal files inside the requested window.

## Success criteria

v0.1 is complete when:

1. Obsidian can create daily notes directly under `journal/` using `templates/journal.md`.
2. The journal template contains exactly these top-level working sections: `## Todo`, `## Notes`, and `## Check-in`.
3. Root non-protected Markdown files remain standalone inbox captures.
4. Direct regular `journal/*.md` files are discovered and indexed as `location_class: "journal"`.
5. Nested journal Markdown files and symlinks are ignored by journal discovery.
6. Invalid direct journal filenames produce warnings.
7. Journal files never affect `brain:inbox` hygiene.
8. `brain:index` emits schema version 2 and includes journal counts.
9. The generated index remains deterministic and does not store journal content or tracking results.
10. `brain:checkin` reports wellbeing scores from valid dated journal notes over a default seven-day window.
11. `brain:checkin -- --to YYYY-MM-DD` anchors the report window deterministically.
12. `brain:checkin -- --json` prints exactly one JSON object to stdout.
13. Wellbeing reporting remains factual and non-clinical.
14. Existing v0 guarantees around determinism, warnings, and atomic index writes remain intact.

## Phase 3 — Journal classification and discovery

### Work

Update the existing classification and discovery modules to support:

```ts
type LocationClass =
  | "protected_root"
  | "journal"
  | "inbox"
  | "wiki"
  | "archive";
```

Classification rules:

```text
exact protected root Markdown file       -> protected_root
direct regular Markdown file in journal/ -> journal
direct non-protected root Markdown file  -> inbox
brain/archive/**/*.md                    -> archive
all other brain/**/*.md                  -> wiki
all other paths                          -> ignored
```

Discovery rules for `brain:index`:

- include protected root Markdown files;
- include direct root inbox Markdown files;
- include direct regular Markdown files under `journal/`;
- include Markdown files under `brain/`;
- ignore nested journal files;
- ignore journal symlinks;
- ignore Markdown outside these scopes, such as `docs/example.md`.

Journal filename validation:

- `journal/YYYY-MM-DD.md` is the only valid journal filename shape;
- valid calendar dates participate in date-based check-in summaries;
- invalid dates produce warnings but still classify as `journal` when the file is direct Markdown under `journal/`;
- invalid filename shapes produce warnings and classify as `journal` when direct Markdown under `journal/`;
- root date-named files remain `inbox`.

### Acceptance

- `journal/2026-06-25.md` classifies as `journal`.
- `journal/2024-02-29.md` classifies as `journal`.
- `journal/2025-02-29.md` classifies as `journal` with an invalid-date warning.
- `journal/2026-13-01.md` classifies as `journal` with an invalid-date warning.
- `journal/2026-6-1.md` classifies as `journal` with an invalid-filename warning.
- `journal/random-note.md` classifies as `journal` with an invalid-filename warning.
- `journal/nested/2026-06-25.md` is ignored.
- `2026-06-25.md` remains `inbox`.
- `brain:inbox` excludes every journal file.

## Phase 4 — Index schema v2

### Work

Upgrade the generated index schema:

```json
{
  "schema_version": 2,
  "documents": []
}
```

A journal record should look like other document records, with:

```json
{
  "path": "journal/2026-06-25.md",
  "location_class": "journal",
  "title": "2026-06-25",
  "frontmatter": {
    "title": "2026-06-25",
    "created": "2026-06-25",
    "kind": "journal"
  },
  "content_hash": "...",
  "warnings": []
}
```
Document warnings should use a generic schema field:

```ts
interface DocumentWarning {
  code: string;
  message: string;
}
```

Warning codes should be stable enough for tests and future tooling. Journal filename and date warnings are document warnings, not frontmatter warnings.

The warning migration boundary is explicit:

- generated schema-v2 index document records use `warnings`;
- `brain:inbox --json` keeps its existing unversioned public `frontmatter_warnings` field;
- changing inbox JSON to generic warnings is out of scope for v0.1.

`warning_count` is the total number of document warnings across every indexed document.

Each document's warnings are emitted in deterministic detection order.

Warning messages must be deterministic and must not include absolute paths, filesystem timestamps, locale-dependent text, or host-specific values.

When a warning refers to a document, use its repository-relative POSIX path.

When a warning refers to a line, use the original 1-based line number.

Warning codes are stable constants, including:

```text
invalid_journal_filename
invalid_journal_date
malformed_frontmatter
duplicate_frontmatter_key
invalid_created_date
```


The index must not contain:

- full journal content;
- parsed todo items;
- parsed notes;
- wellbeing values;
- check-in summaries;
- derived averages;
- task history;
- filesystem modification timestamps.

Summary output should include `journal` in `counts_by_location_class`.

`counts_by_location_class` should emit all five keys even when the count is zero:

```json
{
  "protected_root": 1,
  "journal": 0,
  "inbox": 2,
  "wiki": 2,
  "archive": 1
}
```

### Acceptance

- `brain:index` writes schema version 2.
- `brain:index -- --json` reports journal counts.
- `counts_by_location_class` emits all five location-class keys, including zero counts.
- Document records use `warnings` for generic document warnings, not `frontmatter_warnings`.
- `brain:inbox --json` remains unchanged and continues to expose `frontmatter_warnings`.
- Documents remain sorted by repository-relative POSIX path using raw code-point ordering.
- Unchanged Markdown and paths produce byte-stable generated index output.
- Atomic-write behavior remains intact.
- Document `content_hash` remains raw-byte SHA-256 hashing over the Markdown file content.
- `warning_count` equals the total number of document warnings across indexed documents.
- Document warnings are emitted in deterministic detection order with stable warning codes.
- Warning messages are deterministic and contain no absolute paths, filesystem timestamps, locale-dependent text, or host-specific values.

## Phase 5 — Check-in parsing library

### Work

Add or update library code for journal dates and tracking.

Journal helpers should:

- list valid date-named direct journal files for check-in reporting;
- ignore invalidly named journals for time-series reporting;
- ignore nested journal files;
- parse canonical dates from filenames;
- distinguish missing journal files from existing journals without valid scores.

Tracking helpers should:

- recognize only the exact H2 heading `## Check-in`;
- start parsing after that heading;
- stop parsing at the next H2 heading;
- parse wellbeing entries using this exact line grammar:

```text
^-[ \t]+wellbeing[ \t]*:[ \t]*(.*?)[ \t]*$
```

- accept only decimal integer values with no trailing content;
- a non-blank value is syntactically numeric only when it matches `[0-9]+`;
- `+3`, `-3`, `3.0`, and `3 # comment` are invalid;
- range validation happens after numeric parsing;
- keep `wellbeing` case-sensitive;
- reject alternate bullet markers such as `*`;
- reject inline comments such as `- wellbeing: 3 # comment`;
- reject prose lines such as `text wellbeing: 3`;
- ignore `Wellbeing` and `WELLBEING`;
- treat blank wellbeing as not recorded and not warning;
- treat missing wellbeing as not recorded and not warning;
- warn for every invalid wellbeing occurrence;
- use the last valid wellbeing occurrence as the final score;
- emit one duplicate warning when more than one valid wellbeing occurrence is present;
- never let an invalid later occurrence erase an earlier valid score;
- use a later valid score when an earlier invalid entry appears.

Accepted lines:

```md
- wellbeing: 3
-   wellbeing: 3
- wellbeing : 3
- wellbeing:    3
```

Rejected lines:

```md
* wellbeing: 3
- Wellbeing: 3
- wellbeing: 3 # comment
text wellbeing: 3
```

### Acceptance

- The parser extracts only values from the `## Check-in` block.
- Entries after a later H2 heading are ignored.
- The parser returns structured warnings without throwing for invalid metric values.
- Duplicate semantics are implemented once: every invalid occurrence warns, the last valid score wins, and multiple valid occurrences produce one duplicate warning.
- Empty score summaries are represented as `null` later in CLI JSON output.

## Phase 6 — `brain:checkin` CLI

### Work

Add:

```text
tools/brain/checkin.ts
```

Add package script:

```json
{
  "scripts": {
    "brain:checkin": "tsx tools/brain/checkin.ts"
  }
}
```

Supported commands:

```text
pnpm brain:checkin
pnpm brain:checkin -- --days 7
pnpm brain:checkin -- --days 30
pnpm brain:checkin -- --to 2026-06-25
pnpm brain:checkin -- --json
```

Default behavior:

```text
Metric: wellbeing
Window: last 7 calendar days, ending today
```

Markdown source:

- `brain:checkin` discovers and reads journal Markdown directly.
- It does not read, require, or trust `data/brain-index.json`.
- The generated index may be stale or absent; journal Markdown remains canonical.

Calendar-date semantics:

- `--to` is inclusive.
- `--days N` means N calendar dates inclusive of `--to`.
- `--days 7 --to 2026-06-25` covers 2026-06-19 through 2026-06-25.
- Date-window arithmetic operates on date-only civil values, not UTC timestamps.
- Tests must use `--to`; they must not depend on the host clock or timezone.
- When `--to` is omitted, "today" is the process-local civil calendar date.
- Default today is derived from local year/month/day components, never UTC date components.
- Check-in window logic uses a dedicated CivilDate helper.
- Check-in window logic must not reuse the inbox-age helper that interprets `created` at UTC midnight.

`--to YYYY-MM-DD` behavior:

- anchors the inclusive final date in the calendar window;
- validates that the value is a real `YYYY-MM-DD` civil date;
- errors to stderr and exits non-zero if invalid.

`--days N` behavior:

- controls calendar window size;
- should reject non-integer or non-positive values;
- should produce errors on stderr and return non-zero.

Flag parsing behavior:

- unknown flags error to stderr and exit non-zero;
- repeated flags error to stderr and exit non-zero;
- silently ignored CLI input is not allowed.

Human output:

```text
Wellbeing check-in — 2026-06-19 to 2026-06-25

2026-06-19  3
2026-06-20  —
2026-06-21  4
2026-06-22  2
2026-06-23  3
2026-06-24  3
2026-06-25  4

Recorded: 6/7 days
Average: 3.2
Range: 2–4
```

Human-mode warnings:

- print to stderr;
- do not corrupt stdout report output;
- do not cause a non-zero exit for recoverable data warnings.

JSON output:

```json
{
  "metric": "wellbeing",
  "scale": {
    "min": 1,
    "max": 5
  },
  "from": "2026-06-19",
  "to": "2026-06-25",
  "calendar_days": 7,
  "days_with_journal": 7,
  "recorded_days": 6,
  "entries": [
    {
      "date": "2026-06-19",
      "path": "journal/2026-06-19.md",
      "score": 3
    }
  ],
  "summary": {
    "count": 6,
    "mean": 3.2,
    "min": 2,
    "max": 4
  },
  "warnings": []
}
```

JSON rules:

- stdout contains exactly one valid JSON object;
- stdout contains no human prose;
- errors go to stderr and return non-zero;
- warnings remain structured in the JSON object and do not fail the command;
- `mean` rounds to one decimal place;
- `summary` is `null` when no valid scores exist.
- `entries` contains only days with a valid wellbeing score.
- Missing journals and journals without valid scores remain distinguishable through `calendar_days`, `days_with_journal`, and `recorded_days`, but are not emitted as individual JSON entries in v0.1.

### Acceptance

- Missing journal file and journal without score are distinct internally.
- Both display as `—` in the human time series.
- `days_with_journal` counts existing valid date-named journal files in the window.
- `recorded_days` counts valid wellbeing scores in the window.
- The CLI scans only valid journal filenames for check-in time series.
- Invalid journal filenames do not become fake dates in reports.
- Metric-content warnings are scoped to valid dated journal files inside the selected report window.
- `journal/random-note.md` and `journal/2026-6-1.md` can warn through `brain:index`, but they do not warn in every `brain:checkin` report.
