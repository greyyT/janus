# Janus v0.1 — Journal-First Daily Capture and Wellbeing Tracking

## Status

Draft implementation sketch.

## Goal

Extend Janus with a journal-first daily workflow.

The user opens today’s daily note in Obsidian as the default entrypoint. That note holds:

- the day’s todo list;
- unstructured notes and captures;
- lightweight project follow-ups;
- random thoughts and links;
- one minimal daily check-in score.

Standalone ideas, blogs, articles, or topics that deserve their own identity still become root Markdown inbox notes. Durable knowledge remains deliberately promoted into `brain/`.

Janus stays Markdown-first:

- journal Markdown is canonical;
- generated indexes are disposable;
- tooling reports and validates structure;
- no database, dashboard, or task-management product is introduced.

---

## Principles

1. **Daily capture should be frictionless.**
   The default capture surface is today’s journal note.

2. **The repository root should remain sparse.**
   Root Markdown files are reserved for standalone inbox captures, not recurring daily notes.

3. **Chronological working memory is separate from durable knowledge.**
   `journal/` stores daily context. `brain/` stores promoted, maintained knowledge.

4. **Tracking should be factual and non-clinical.**
   Janus records subjective scores and summarizes them. It does not diagnose, predict, prescribe, or classify mental-health status.

5. **The note format should be extensible without migration.**
   Future metrics can be added to the existing `## Check-in` section.

`journal/` is a flat directory in v0.1.

Do not use year/month nesting such as:

```text
journal/2026/06/25.md
journal/2026/06/2026-06-25.md
```

A single daily note per day is small enough for a flat directory. Adding hierarchy before it becomes necessary makes Obsidian navigation and tooling less predictable.

## Note taxonomy

| Location                              | Purpose                                      | Authority                                 |
| ------------------------------------- | -------------------------------------------- | ----------------------------------------- |
| Root `*.md` excluding protected files | Standalone unstructured inbox captures       | Low; may be rough or incomplete           |
| `journal/YYYY-MM-DD.md`               | Daily working memory, tasks, notes, check-in | Personal chronological record             |
| `brain/**/*.md`                       | Promoted durable knowledge                   | Maintained and deliberate                 |
| `brain/archive/**/*.md`               | Archived durable knowledge                   | Historical                                |
| `data/`                               | Generated indexes                            | Disposable, lower authority than Markdown |

Examples:

```text
onibi-transfer-guardrail-idea.md     → root inbox
blog-agent-memory-architecture.md    → root inbox
journal/2026-06-25.md                → journal
brain/projects/janus/vision.md       → wiki
brain/archive/old-system.md          → archive
```

## Journal file contract

Daily notes live directly under `journal/`:

```text
journal/YYYY-MM-DD.md
```

The filename date is the canonical journal date.

```text
journal/2026-06-25.md  → valid journal date
journal/2024-02-29.md  → valid journal date
journal/2025-02-29.md  → journal with invalid-date warning
journal/2026-13-01.md  → journal with invalid-date warning
journal/2026-6-1.md    → journal with invalid-filename warning
journal/random-note.md → journal with invalid-filename warning
```

Rules:

- only direct regular Markdown files inside `journal/` are journal documents;
- symlinks are ignored;
- nested files under `journal/` are ignored in v0.1;
- all direct `journal/*.md` files receive `location_class: "journal"`;
- only valid date-named files participate in date-based check-in summaries;
- invalid journal filenames produce warnings rather than silently disappearing.

A root-level date file is not a journal:

```text
2026-06-25.md → inbox
journal/2026-06-25.md → journal
```

This keeps the location contract explicit and avoids accidental root clutter.

## Daily workflow

### Start of day

In Obsidian, open or create:

```text
journal/YYYY-MM-DD.md
```

This is the daily entrypoint.

Add tasks to `## Todo`. Put anything lightweight, contextual, or unfinished under `## Notes`.

### During the day

Use the journal note for:

- quick work follow-ups;
- personal reminders;
- loose ideas;
- links worth revisiting;
- random observations;
- short decision records;
- rough notes from conversations;
- context related to today’s tasks.

Create a standalone root inbox note only when the content deserves its own title, source context, or likely promotion path.

Examples:

```text
blog-agent-memory-architecture.md
react-server-components-caching.md
onibi-transfer-validation-idea.md
```

Reference that note from the daily journal when useful:

```md
- Created [[blog-agent-memory-architecture]] after reading the article.
- [ ] Extract the useful ideas into durable Janus knowledge.
```

### End of day

- Mark completed todo items.
- Leave incomplete tasks in place.
- Manually copy forward only tasks still worth doing tomorrow.
- Record the wellbeing score when appropriate.

There is no automated task rollover in v0.1.

Manual carry-forward is intentional: it forces a small daily prioritization decision and avoids introducing task-manager behavior prematurely.

```text
templates/journal.md
```

```md
---

```

Janus does not need a command that opens Obsidian or creates the note. Obsidian owns the interaction; Janus owns the repository structure and reporting behavior.

## Journal section semantics

### `## Todo`

The active task list for that date.

```md
## Todo

- [ ] Review PR #482
- [ ] Investigate malformed QR merchant identifiers
- [x] Send product feedback
```

Rules:

- use ordinary Markdown checkbox syntax;
- completed tasks remain in the note;
- unfinished tasks are manually copied forward when still relevant;
- do not add task metadata, priorities, due dates, or workflows in v0.1;
- do not aggregate todos globally in v0.1.

### `## Notes`

The unstructured capture surface for the day.

```md
## Notes

- Need to verify whether the QR parser accepts malformed merchant IDs.
- Idea: Janus should separate chronological working memory from durable knowledge.
- Blog: https://example.com/agent-memory
- Created [[blog-agent-memory-architecture]] after reading the article.
- Follow up with Tino about React architecture.

Today was fragmented by meetings, but the implementation block after lunch was productive.
```

`## Notes` is deliberately unstructured.

It may contain:

- bullets;
- prose;
- links;
- code snippets;
- headings below level two;
- references to other notes;
- rough task context;
- incomplete ideas.

Janus v0.1 does not parse or interpret the contents of `## Notes`.

### `## Check-in`

The structured personal-metric section.

```md
## Check-in

- wellbeing: 3
```

Only this section is parsed by tracking tooling.

---

## First tracking metric: wellbeing

The first supported metric is:

```text
wellbeing: 1–5
```

Suggested interpretation:

```text
1 → very difficult day
2 → low / strained
3 → okay / neutral
4 → good
5 → very good
```

This is a subjective self-reporting scale, not a mental-health assessment.

Janus must not:

- diagnose mental-health conditions;
- label a user as healthy or unhealthy;
- infer risk;
- interpret a score trend as clinically meaningful;
- prescribe actions based on a score.

The tracker should only report recorded values, coverage, average, minimum, and maximum.

---

## Generic metric format

The check-in format should support future metrics without changing existing journal notes.

```md
## Check-in

- wellbeing: 3
- sleep_quality: 4
- energy: 2
- stress: 4
```

v0.1 recognizes only `wellbeing`.

Metric definitions should be held in code:

```ts
interface TrackingMetricDefinition {
  key: string;
  label: string;
  min: number;
  max: number;
}

const TRACKING_METRICS: Record<string, TrackingMetricDefinition> = {
  wellbeing: {
    key: "wellbeing",
    label: "Wellbeing",
    min: 1,
    max: 5,
  },
};
```

Adding a future metric should require adding one definition to this registry, not changing the Markdown structure or migrating prior notes.

## Check-in parsing contract

The parser recognizes only the exact H2 section:

```md
## Check-in
```

It starts after that heading and stops at the next H2 heading.

Example:

```md
## Check-in

- wellbeing: 3
- sleep_quality: 4

## Later reflection

- wellbeing: 1
```

Only the first `wellbeing: 3` is relevant to tracking. The later entry is part of free-form content and must be ignored.

Supported entry format:

```text
- metric_name: integer_value
```

For wellbeing:

```md
- wellbeing: 3
```

Behavior:

| Input                              | Result                                         |
| ---------------------------------- | ---------------------------------------------- |
| `- wellbeing: 3`                   | Valid score                                    |
| `- wellbeing:`                     | Not recorded; no warning                       |
| No wellbeing entry                 | Not recorded; no warning                       |
| `- wellbeing: 0`                   | Warning; out of range                          |
| `- wellbeing: 6`                   | Warning; out of range                          |
| `- wellbeing: okay`                | Warning; not an integer                        |
| Two valid wellbeing entries        | Last valid value wins; warning                 |
| Valid then invalid wellbeing entry | Valid score remains; warning for invalid value |
| Invalid then valid wellbeing entry | Valid score is used; warning for invalid value |

Keys are case-sensitive in v0.1:

```text
wellbeing → recognized
Wellbeing → ignored
WELLBEING → ignored
```

This avoids ambiguous normalization behavior in the first implementation.

## Tooling changes

### Location classes

Extend the existing type:

```ts
type LocationClass =
  | "protected_root"
  | "journal"
  | "inbox"
  | "wiki"
  | "archive";
```

Classification rules:

```text
exact protected root Markdown file       → protected_root
direct regular Markdown file in journal/ → journal
direct non-protected root Markdown file  → inbox
brain/archive/**/*.md                    → archive
all other brain/**/*.md                  → wiki
all other paths                          → ignored
```

Protected root filenames remain the existing exact case-sensitive set:

```text
AGENTS.md
CLAUDE.md
README.md
CONTRIBUTING.md
LICENSE.md
CHANGELOG.md
CODE_OF_CONDUCT.md
SECURITY.md
```

### Discovery rules

`brain:index` discovers:

1. protected root Markdown files;
2. ordinary root inbox Markdown files;
3. direct regular Markdown files under `journal/`;
4. Markdown files under `brain/`.

It does not discover:

```text
journal/nested/example.md
journal/symlink.md
docs/example.md
brainx/example.md
source-code Markdown files outside brain/
```

### Index schema

Upgrade the generated index schema:

```json
{
  "schema_version": 2,
  "documents": []
}
```

A valid journal record:

```json
{
  "path": "journal/2026-06-25.md",
  "location_class": "journal",
  "title": "2026-06-25",
  "frontmatter": {
    "title": "2026-06-25",
    "created": "2026-06-25",
    "kind": "journal"
  },
  "content_hash": "...",
  "warnings": []
}
```

Document warnings use stable warning codes and deterministic messages. Journal filename warnings are document warnings, not frontmatter warnings. `brain:inbox -- --json` keeps its existing `frontmatter_warnings` contract.

The index must not contain:

- full journal content;
- parsed todo items;
- parsed free-form notes;
- check-in values;
- derived averages;
- task history;
- filesystem modification timestamps.

Markdown remains canonical. The index remains deterministic and disposable.

### `brain:inbox`

Existing inbox behavior remains narrow.

```text
pnpm brain:inbox
pnpm brain:inbox -- --json
```

It inspects only direct root inbox Markdown files.

It must:

- exclude `journal/`;
- exclude protected root files;
- preserve root inbox hygiene thresholds;
- preserve its existing JSON contract;
- continue reporting age based on frontmatter `created` or filesystem modification time.

Example:

```text
janus/
├── blog-agent-memory.md
├── project-idea.md
└── journal/
    ├── 2026-06-24.md
    ├── 2026-06-25.md
    └── 2026-06-26.md
```

Expected inbox result:

```text
Inbox count: 2
Hygiene: healthy
```

### `brain:index`

Extend the existing index command:

```text
pnpm brain:index
pnpm brain:index -- --json
```

Summary output gains journal counts:

```json
{
  "output_path": "data/brain-index.json",
  "document_count": 14,
  "counts_by_location_class": {
    "protected_root": 1,
    "journal": 8,
    "inbox": 2,
    "wiki": 2,
    "archive": 1
  },
  "warning_count": 0
}
```

`counts_by_location_class` emits all five location-class keys even when a count is zero. `warning_count` is the total number of document warnings across indexed documents.

The command keeps the existing deterministic ordering and atomic-write requirements.

### New `brain:checkin` command

Add:

```text
pnpm brain:checkin
pnpm brain:checkin -- --days 7
pnpm brain:checkin -- --days 30
pnpm brain:checkin -- --to 2026-06-25
pnpm brain:checkin -- --json
```

Default:

```text
Metric: wellbeing
Window: last 7 calendar days, ending today
```

When `--to` is omitted, today is the process-local civil calendar date. Date-window arithmetic uses date-only civil values, not UTC timestamp semantics.

`brain:checkin` discovers and reads journal Markdown directly. It does not read, require, or trust `data/brain-index.json`.

Example human output:

```text
Wellbeing check-in — 2026-06-19 to 2026-06-25

2026-06-19  3
2026-06-20  —
2026-06-21  4
2026-06-22  2
2026-06-23  3
2026-06-24  3
2026-06-25  4

Recorded: 6/7 days
Average: 3.2
Range: 2–4
```

Definitions:

```text
—                 no valid wellbeing score recorded
days_with_journal journal file exists for that date
recorded_days     valid wellbeing score exists for that date
```

A missing journal file and a journal without a score are distinct, but neither is an error.

`brain:checkin` scans only valid journal filenames:

```text
journal/YYYY-MM-DD.md
```

It ignores:

- root inbox notes;
- `brain/` notes;
- invalidly named journal files;
- nested journal files.

Invalid journal files remain visible to `brain:index` through warnings, but they do not become fake dates in a check-in time series.

Metric-content warnings are scoped to valid dated journal files inside the selected report window.

### Check-in JSON contract

```json
{
  "metric": "wellbeing",
  "scale": {
    "min": 1,
    "max": 5
  },
  "from": "2026-06-19",
  "to": "2026-06-25",
  "calendar_days": 7,
  "days_with_journal": 7,
  "recorded_days": 6,
  "entries": [
    {
      "date": "2026-06-19",
      "path": "journal/2026-06-19.md",
      "score": 3
    }
  ],
  "summary": {
    "count": 6,
    "mean": 3.2,
    "min": 2,
    "max": 4
  },
  "warnings": []
}
```

Rules:

- JSON mode prints one valid JSON object to stdout;
- JSON mode prints no human prose to stdout;
- errors go to stderr and return non-zero;
- warnings remain structured output and do not fail the command;
- `mean` rounds to one decimal place;
- if no valid scores exist, `summary` is `null`.
- `entries` contains only days with a valid wellbeing score;
- missing journals and journals without valid scores remain distinguishable through `calendar_days`, `days_with_journal`, and `recorded_days`, but are not emitted as individual JSON entries in v0.1;

```text

    ├── classify.ts
    ├── documents.ts
    ├── journal.ts
```

Responsibilities:

```text
classify.ts
  Extends location classification with journal.

journal.ts
  Finds direct journal Markdown files, validates YYYY-MM-DD filenames,
  parses canonical journal dates.

tracking.ts
  Extracts the exact ## Check-in section, validates metric values,
  resolves duplicate entries, and builds date-window summaries.

checkin.ts
  Parses CLI arguments and formats output only.

documents.ts
  Reuses existing title, frontmatter, hash, and document-record behavior.

output.ts
  Formats human-readable and JSON check-in output.
```

No new runtime dependencies are required.

## Explicit non-goals

Do not implement in v0.1:

- automatic task rollover;
- global task aggregation;
- task due dates, priorities, or status systems;
- reminders or notifications;
- a dashboard;
- graphs;
- a database;
- Obsidian plugin code;
- automatic journal-note creation from CLI;
- automatic promotion to `brain/`;
- automatic extraction from `## Notes`;
- multi-metric reporting;
- mood prediction;
- clinical interpretation;
- embeddings, semantic search, backlinks, or knowledge graphs.

---

## Done definition

v0.1 is complete when:

1. Daily notes live in `journal/` as `YYYY-MM-DD.md`.
2. Obsidian Daily Notes opens or creates the journal note as the user’s default daily entrypoint.
3. The journal template contains exactly Todo, Notes, and Check-in as its top-level working sections.
4. `## Notes` is the daily unstructured capture area.
5. Root Markdown files remain standalone inbox captures, except protected repository files.
6. Journal files never affect root inbox hygiene.
7. `brain:index` indexes direct journal Markdown files with `location_class: "journal"`.
8. Invalid journal filenames remain visible as warnings.
9. `pnpm brain:checkin -- --days 7` reports valid wellbeing scores from dated journal notes.
10. Wellbeing reporting remains factual and non-clinical.
11. Tracking data remains canonical in Markdown and is not persisted as derived index content.
12. Existing v0 determinism, warning, and atomic-index-write guarantees remain intact.
