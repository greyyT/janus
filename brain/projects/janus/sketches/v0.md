# Janus v0 — Root-Drop Inbox Model

## Core rule

Janus has no `brain/inbox/` directory.

Instead, every Markdown file placed directly in the repository root is treated as an inbox item unless it is a protected default file.

```text
janus/
├── AGENTS.md          # protected
├── CLAUDE.md          # protected
├── README.md          # protected
├── CONTRIBUTING.md    # protected, optional
├── LICENSE.md         # protected, optional
│
├── idea-agent-memory.md        # inbox note
├── onibi-transfer-bug.md       # inbox note
├── notes-from-meeting.md       # inbox note
│
├── brain/             # durable wiki
├── registry/
├── tools/
├── data/
└── .agents/
```

A root-level Markdown file is not automatically considered durable knowledge. It is a captured thought waiting to be processed.

---

## Why this is preferable

This removes the capture friction of deciding:

- which folder the note belongs in;
- whether it is a project, decision, pattern, investigation, or concept;
- whether it deserves metadata;
- whether it is worth keeping at all.

The workflow becomes:

```text
Have a thought
  ↓
Drop one Markdown file into Janus root
  ↓
Janus treats it as an inbox item
  ↓
Later: discard, archive, or promote it into the wiki
```

The physical repository remains simple while the semantic distinction remains clear.

## Protected root files

The indexer should explicitly exclude these files from inbox handling:

```text
AGENTS.md
CLAUDE.md
README.md
CONTRIBUTING.md
LICENSE.md
CHANGELOG.md
CODE_OF_CONDUCT.md
SECURITY.md
```

Everything else matching this rule is an inbox note:

```text
/*.md
```

Only Markdown files directly under the repository root count as inbox items.

These do not count as inbox items:

```text
brain/**/*.md
tools/**/*.md
docs/**/*.md
.agents/**/*.md
.claude/**/*.md
```

This prevents an accidental distinction where every Markdown file in the entire repository becomes an inbox note.

## Root inbox note rules

Inbox notes should have almost no required structure.

This is valid:

```md
# Janus idea

What if project pages become the primary way agents discover architecture,
while semantic search only activates when lexical search is weak?

Questions:

- Do we need embeddings early?
- Should project indexes have a freshness score?
```

This is also valid:

```md
I found a weird issue in transfer validation.

Need to check whether the balance is being read before token resolution.
```

Do not require frontmatter for root inbox notes.

Optional frontmatter is allowed when useful:

```md
---
project: onibi
created: 2026-06-24
---

# Transfer validation concern

...
```

But Janus should never make metadata a requirement for capture.

## Inbox note lifecycle

```text
Root Markdown note
  ↓
Temporary capture
  ↓
Review
  ├─ Delete: no durable value
  ├─ Archive: useful historical record only
  ├─ Keep at root: still active or unresolved
  └─ Promote: becomes a typed durable wiki note
```

### Keep at root

A note may remain at the root while it is:

- a current thought;
- incomplete;
- an active investigation;
- waiting for validation;
- a task-specific scratchpad;
- not yet worth organizing.

There should be no pressure to process every root note immediately.

### Promote

A note should be promoted when it records durable knowledge, such as:

- a technical decision;
- a recurring implementation pattern;
- an architecture map;
- a reliable project convention;
- a debugging lesson likely to recur;
- an explanation that will help future work.

Promotion means:

1. Move the file into `brain/<type>/`.
2. Add the required frontmatter.
3. Rename it into a stable slug if needed.
4. Add links to relevant projects, systems, or decisions.
5. Update the relevant project `INDEX.md`.
6. Rebuild the derived search index.

Example:

```text
janus/onibi-transfer-bug.md
```

becomes:

```text
janus/brain/patterns/validate-token-before-balance-check.md
```

or:

```text
janus/brain/projects/onibi/decisions/transfer-validation-order.md
```

depending on what the note actually represents.

## Inbox search behavior

Root inbox notes should be searchable, but treated differently from promoted wiki notes.

Search result authority should follow this order:

```text
Repository source code and repository-local documentation
  >
Promoted Janus wiki notes
  >
Root inbox notes
  >
Generated summaries, indexes, embeddings, and harness auto-memory
```

Inbox items are useful for discovering current work and unresolved thoughts, but they should not override durable or repository-owned facts.

Suggested ranking behavior:

```text
Exact project/path/title matches
  >
Promoted notes with accepted/active status
  >
Promoted notes with relevant links
  >
Root inbox notes with direct keyword matches
  >
Semantic-only matches
```

A root note may be returned with an explicit marker:

```text
[inbox / unverified]
onibi-transfer-bug.md
Matched heading: "Balance read before token resolution"
```

This makes uncertainty visible to both you and the coding harness.

## Updated `AGENTS.md` rules

```md
# Janus

Janus is a Markdown-first personal engineering knowledge system.

## Knowledge locations

- `brain/` contains promoted, durable wiki knowledge.
- Non-default Markdown files placed directly in the repository root are inbox notes.
- Root inbox notes are temporary and may be incomplete, incorrect, or unverified.
- Source code and repository-local documentation remain authoritative for code behavior.

## Source-of-truth hierarchy

1. Source code and repository-local documentation.
2. Promoted Janus wiki notes under `brain/`.
3. Root inbox notes.
4. Generated indexes, embeddings, summaries, and harness auto-memory.

## Before making a technical decision

1. Read `brain/HOME.md`.
2. Identify the relevant project page under `brain/projects/`.
3. Search both `brain/` and root inbox notes.
4. Read original Markdown source files, not only search snippets.
5. Verify code-specific claims in the relevant source repository.

## Capturing knowledge

- For rough thoughts, create a Markdown file directly in the Janus root.
- Do not require frontmatter, tags, templates, or classification for root notes.
- Do not store durable technical facts only in a root inbox note.
- Promote durable and verified knowledge into `brain/`.
- Delete low-value or obsolete inbox notes rather than organizing everything.

## Promotion rules

When promoting a root note:

1. Determine whether it is a project, system, decision, pattern, playbook, concept, or investigation.
2. Move it under the correct `brain/` location.
3. Add frontmatter and a stable title.
4. Link it from the relevant project index.
5. Preserve important history; archive rather than silently erasing superseded decisions.

## Do not

- Do not treat inbox notes as verified facts.
- Do not create a separate `brain/inbox/` folder.
- Do not index secrets, keys, credentials, or sensitive production data.
- Do not duplicate repository-owned architecture documentation.
- Do not load the whole knowledge base into context by default.
```

## Suggested root-file naming

Use loose but meaningful names.

Good:

```text
janus-memory-ideas.md
onibi-transfer-validation.md
meeting-notes-tino-2026-06-24.md
bug-sse-ordering.md
thoughts-on-agent-context.md
```

Avoid generic accumulating files:

```text
notes.md
ideas.md
random.md
todo.md
```

Each root file should represent one active train of thought where possible.

It does not need to be perfectly atomic. The promotion process will split and refine it later.

---

## Root hygiene rule

The repository root should remain small enough to scan visually.

Suggested soft limits:

```text
0–10 root inbox notes: healthy
11–20 root inbox notes: review soon
20+ root inbox notes: process, delete, archive, or promote
```

This is not a hard technical constraint. It is a signal that the capture surface is becoming a dumping ground rather than an active working queue.

## Search implementation change

The search index should store a `location_class` field:

```text
location_class:
  - protected_root
  - inbox
  - wiki
  - archive
```

Example document record:

```text
path: onibi-transfer-validation.md
location_class: inbox
kind: null
status: unverified
authority_weight: 0.35
```

Promoted note:

```text
path: brain/patterns/validate-before-execution.md
location_class: wiki
kind: pattern
status: active
authority_weight: 1.0
```

Archived note:

```text
path: brain/archive/initial-transfer-validation-notes.md
location_class: archive
status: superseded
authority_weight: 0.15
```

The index should automatically infer `location_class` from file path. Root notes should not need metadata to be recognized as inbox items.

## Small tooling surface

```text
pnpm brain:inbox
pnpm brain:search -- "transfer validation"
pnpm brain:promote -- onibi-transfer-validation.md --type pattern
pnpm brain:links --check
pnpm brain:index
```

These are support scripts for the harness and repository maintenance. They are not a custom Janus CLI product.

`brain:inbox` should list:

- root inbox files;
- age;
- optional project metadata;
- whether the note is linked anywhere;
- whether it has likely duplicate content in `brain/`.

---

## Principle

The root is optimized for capture.

`brain/` is optimized for retrieval and durability.

The transition from root to `brain/` is the deliberate act of deciding:

> This thought is worth preserving as part of my engineering memory.
