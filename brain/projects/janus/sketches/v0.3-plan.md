# Janus v0.3 Implementation Plan

## Status

Planning document for implementing [`v0.3.md`](./v0.3.md).

## Locked decisions

1. Implement all v0.3 features as one staged v0.3 plan, not as separate versions and not as one giant cutover.
2. Add root `backlog.md` as the canonical queue of unresolved, uncommitted tasks.
3. Treat `backlog.md` as a reserved protected root Markdown file, not an inbox note.
4. Create `backlog.md` during implementation. Runtime commands may require it to exist after v0.3 is installed.
5. Add harness-native workflows:
   - `/add-task`
   - `/checkin`
   - `/checkout`

6. Add deterministic task CLIs under `pnpm brain:task:*`; slash workflows orchestrate and call CLIs after user approval.
7. Use namespaced task scripts, not verb aliases:
   - `brain:task:add`
   - `brain:task:list`
   - `brain:task:move`
   - `brain:task:checkin`
   - `brain:task:checkout`

8. Mutating task CLIs support `--dry-run --json` previews.
9. Task move commands use semantic locations such as `backlog`, `today`, and `journal:YYYY-MM-DD`, not raw paths.
10. Use stable sequential task IDs: `J-001`, `J-002`, and so on.
11. The ID allocation source of truth is the `backlog.md` HTML comment:

    ```md
    <!-- janus-backlog: next_task_id=7 -->
    ```

12. `/add-task` always writes new tasks to `backlog.md`; committing tasks to today happens through `/checkin` or explicit task movement.
13. Use one flat backlog. Do not add Ready/Later/Waiting categories.
14. Use `estimate: quick|medium|large` as the parseable estimate format. Extra detail belongs in task context or updates.
15. Preserve rough task context beneath each task block as user-owned Markdown.
16. An active task is the complete top-level Markdown checkbox block containing `[J-###]`.
17. Each open task ID exists in exactly one active location:
    - `backlog.md`; or
    - one `journal/YYYY-MM-DD.md` `## Todo` section.

18. `/checkin` is agent-led. The user invokes it every morning; the agent reconstructs context, recommends the day, asks for approval/correction, then applies the plan.
19. `/checkin` creates today's journal note from `templates/journal.md` when missing.
20. `/checkin` scans only the latest prior journal containing unfinished active tasks.
21. Unfinished tasks remain in the prior journal after `/checkout`; the next `/checkin` reconciles them.
22. Carryover dispositions during `/checkin` are:
    - recommit today;
    - return to backlog;
    - split;
    - cancel;
    - mark completed if finished after checkout.

23. Carryover decisions are recorded by appending factual audit entries to the prior journal's `## Checkout`.
24. Splitting a carryover task creates fresh IDs, and each child task may be placed either in today's `## Todo` or `backlog.md`.
25. `/checkin` applies its morning changes through one structured CLI apply plan, previewed with `--dry-run --json` before approval.
26. `/checkout` does not move unfinished tasks back to `backlog.md`.
27. `/checkout` marks completed task blocks `[x]`, leaves unfinished task blocks `[ ]`, and writes the `## Checkout` reflection.
28. Completed tasks remain as full completed task blocks in the journal.
29. `/checkout` uses CLI-owned writes for task completion and checkout reflection.
30. `/checkout` uses harness guarded edits for approved root inbox or durable `brain/` knowledge changes.
31. `/checkout` may identify durable `brain/` edits, but durable knowledge edits require an explicit approval gate.
32. `/checkout` may create root inbox captures as part of the daily digest.
33. Calendar support is optional but implemented in v0.3.
34. `/checkin` must work without calendar configuration by asking for missing constraints.
35. Calendar configuration uses one fixed local export:

    ```text
    .janus/calendar/primary.ics
    .janus/calendar/config.json
    ```

36. Use a focused, verified iCalendar dependency if it fits Node 20 and ESM. Wrap it behind Janus calendar library code.
37. Support a single calendar export. Do not add multi-calendar management.
38. Delete the unused `pnpm brain:checkin` wellbeing report command.
39. Remove all current wellbeing reporting tooling. `/checkout` records wellbeing as journal content; v0.3 does not ship a wellbeing reporting CLI.
40. Do not add task state, calendar details, journal todos, daily reflections, or digest outcomes to `data/brain-index.json`.
41. Update both the v0.3 sketch and this implementation plan so future agents do not follow stale lifecycle rules.

`backlog.md` initial shape:

```md
# Backlog

<!-- janus-backlog: next_task_id=1 -->
```

The backlog parser must support:

```text
top-level Markdown checkbox line containing [J-###]
+ every indented line that follows
+ until the next top-level task block or end of file
```

Parser behavior:

- Preserve unknown indented Markdown.
- Reject duplicate active task IDs.
- Reject task lines with multiple task IDs.
- Reject malformed task IDs.
- Reject missing or invalid `next_task_id` comment.
- Reject a counter lower than max existing task ID + 1.
- Treat indented nested checkboxes as task body content, not active tasks.

### Acceptance

- `backlog.md` is classified as `protected_root`, not `inbox`.
- `pnpm brain:inbox -- --json` does not return `backlog.md`.
- The backlog parser preserves user Markdown exactly outside intentional task edits.
- Tests cover valid tasks, malformed IDs, duplicate IDs, nested checkboxes, invalid counters, and deterministic ordering.

## Stage 3 — Task CLI foundation

### Work

Create namespaced task scripts in `package.json`:

```json
{
  "brain:task:add": "tsx tools/brain/task-add.ts",
  "brain:task:list": "tsx tools/brain/task-list.ts",
  "brain:task:move": "tsx tools/brain/task-move.ts",
  "brain:task:checkin": "tsx tools/brain/task-checkin.ts",
  "brain:task:checkout": "tsx tools/brain/task-checkout.ts"
}
```

Shared rules:

- All mutating commands support `--dry-run --json`.
- Mutating commands validate all affected task IDs before writing.
- Semantic locations are used:

  ```text
  backlog
  today
  journal:YYYY-MM-DD
  ```

- Writes are atomic where practical.
- Human output is concise; JSON output is stable for slash workflows and tests.

### Acceptance

- `brain:task:list -- --json` reports active tasks from `backlog.md` and journal task locations.
- `brain:task:add -- --dry-run --json` previews the next ID without mutating `backlog.md`.
- `brain:task:move -- --dry-run --json --from backlog --to today J-001` previews exact source and destination changes.
- Tests prove dry-run does not mutate files.

## Stage 4 — `/add-task`

### Work

Create:

```text
.claude/commands/add-task.md
```

`/add-task` behavior:

1. Ask for task title.
2. Ask whether there is a deadline, trigger, or real time constraint.
3. Ask estimate as `quick`, `medium`, or `large`.
4. Ask whether it is blocked.
5. Ask whether to preserve rough context.
6. Call `brain:task:add` after enough information is gathered.

Rules:

- Only the title is required.
- `/add-task` always writes to `backlog.md`.
- It does not decide whether the task is ready, important, or appropriate for today.

### Acceptance

- A simple task can be added with minimal interaction.
- A vague or large task gives the user one chance to preserve context without forcing a full plan.
- The resulting task receives a stable ID and increments the backlog counter.

## Stage 6 — Agent-led `/checkin`

### Work

Create:

```text
.claude/commands/checkin.md
tools/brain/lib/daily-workflow.ts
tools/brain/lib/daily-workflow.test.ts
```

`/checkin` reads:

1. Today's journal, creating it if missing.
2. The latest prior journal containing unfinished active tasks.
3. The prior journal's `## Checkout`, especially `handoff` and `next_step`.
4. Remaining tasks in `backlog.md`.
5. Calendar availability from `brain:calendar`, when configured.
6. User-stated constraints not represented in Janus or the calendar.

`/checkin` is agent-led:

1. The agent reconstructs morning context.
2. The agent recommends a daily plan with reasons.
3. The user approves or corrects the plan once.
4. The agent previews a single structured apply plan via `brain:task:checkin -- --dry-run --json`.
5. After approval, the agent applies the same plan.

Carryover dispositions:

- recommit today;
- return to backlog;
- split;
- cancel;
- mark completed if finished after checkout.

Carryover decisions are appended to the prior journal's `## Checkout`.

### Acceptance

- `/checkin` does not require manual pre-creation of today's journal.
- `/checkin` works without configured calendar input.
- `/checkin` uses the latest prior journal only for carryover recovery.
- Recommitted tasks move into today's `## Todo`.
- Returned tasks move to `backlog.md`.
- Split tasks receive fresh IDs and per-child destinations.
- Cancelled tasks do not return to backlog.
- All planned mutations are previewable as JSON before applying.

## Stage 8 — `/checkout`

### Work

Create:

```text
.claude/commands/checkout.md
```

`/checkout` behavior:

1. Ask which active tasks were completed.
2. Mark completed task blocks `[x]` using `brain:task:checkout`.
3. Leave unfinished task blocks `[ ]` in today's journal.
4. Ask fixed reflection prompts:
   - wellbeing;
   - worked;
   - improve;
   - handoff;
   - next_step.
5. Write `## Checkout` via CLI-owned journal task/reflection writes.
6. Perform daily-note digest review.
7. Use guarded harness edits for approved root inbox or durable `brain/` changes.

Task movement rule:

```text
/checkout never moves unfinished task blocks back to backlog.md.
```

### Acceptance

- Completed tasks remain as full checked task blocks in the journal.
- Unfinished tasks remain unchecked for next `/checkin`.
- `## Checkout` contains reflection, handoff, next step, task decisions, and digest audit.
- `/checkout` can run even if no tasks were completed.

## Stage 9 — Daily-note digest during checkout

### Work

During `/checkout`, review:

- `## Notes`;
- task outcomes;
- decisions;
- references and links;
- unresolved ideas;
- checkout handoff and next step.

Disposition rules:

| Finding | Action |
| --- | --- |
| Open work that survives today | Add/update `backlog.md` through task CLIs |
| Rough standalone idea, link, or reference | Create a root inbox capture |
| Clear correction to durable knowledge | Propose a `brain/` edit and require explicit approval |
| Ambiguous or purely temporal context | Keep in the journal only |

Source trail format:

```md
Source: journal/YYYY-MM-DD.md — Notes
```

### Acceptance

- Every checkout reviews the daily note.
- Not every daily note creates an inbox capture or durable note.
- Root inbox captures created from checkout include a source trail.
- Durable `brain/` edits are approval-gated.
- Generated index refresh happens after approved durable knowledge edits.

## Non-goals

- automatic next-day task rollover;
- automatic scheduling;
- opaque priority scores;

- Kanban boards;
