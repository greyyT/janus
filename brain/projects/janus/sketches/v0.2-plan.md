# Janus v0.2 Implementation Plan

## Status

Planning document for implementing [`v0.2.md`](./v0.2.md).

## Locked decisions

1. Implement v0.2 as a harness-native `/digest` workflow, not as a new Janus application surface.
2. Create `.claude/commands/digest.md` as the interactive agent command.
3. Create `brain/projects/janus/digest-workflow.md` as durable user-facing documentation.
4. Add a local digest checklist at `.janus/digest-ledger.md`.
5. Add `pnpm brain:digest:init` as an initializer command that creates `.janus/digest-ledger.md` from the authoritative inbox queue.
6. `brain:digest:init` must preserve an existing ledger by default and report that it is being resumed.
7. `pnpm brain:digest:init -- --reset` explicitly overwrites `.janus/digest-ledger.md`.
8. `/digest` should guide the agent to use the existing ledger by default. Reset must be an explicit user instruction.
9. The ledger uses one Markdown checklist section.
10. Checklist lines use exact inbox paths, with optional disposition suffixes:

   ```md
   - [ ] note-a.md
   - [x] note-b.md — clarified
   - [x] note-c.md — parked
   - [x] note-d.md — keep raw
   - [x] note-e.md — proposed discard
   ```

11. `[x]` means no further teacher/retrieval work is required for this digest run. It does not mean the final approval gate has approved source changes, and it does not mean approved edits have been applied.
12. The suffix records the note's current disposition. `proposed discard` remains unapproved until the final approval gate.
13. Before approval, the agent may modify only `.janus/digest-ledger.md`.
14. Before approval, the agent must not modify inbox notes, journal notes, brain notes, archive notes, templates, source code, or project documentation.
15. The ledger is a progress checkpoint, not a full resume database. Detailed reasoning remains in the conversation ledger during the active digest.
16. `.janus/digest-ledger.md` is gitignored with a root-anchored ignore rule: `/.janus/digest-ledger.md`.
17. Update `brain/projects/janus/sketches/v0.2.md` as a deliberate design revision because the original sketch forbids workflow-state files.
18. Archive approved promoted root captures under `brain/archive/inbox/`.
19. If an archive destination already exists, stop and ask the user for a destination. Never overwrite or auto-merge archived captures.
20. If an inbox note conflicts with durable `brain/` knowledge and the user does not resolve the conflict, default to `park_for_later` and leave the source note in root inbox.
21. Do not use retrieval scout subagents in v0.2. The coordinator performs retrieval itself.
22. Do not add embeddings, BM25, RAG, vector databases, background queues, new index schemas, automatic tags, automatic task promotion, or an agent-test runner.

## Pushback applied to the sketch

The original v0.2 sketch keeps all digest state in conversation and explicitly forbids writing a workflow-state file. The implementation will revise that design because long multi-file digests need a low-friction external progress checkpoint.

The checkpoint is intentionally narrow:

- It is local operational state, not durable knowledge.
- It records file progress only, not full evidence or user-answer history.
- It exists so a future hook can detect whether the agent recorded an outcome before moving to another inbox note.
- It does not replace the in-conversation digest ledger required for reasoning, retrieval evidence, open gaps, and organization decisions.

The initializer must support resume instead of erasing progress:

```text
No ledger exists          -> create it
Ledger exists             -> preserve it and report that it is being resumed
Explicit --reset          -> overwrite it
```

At startup, `/digest` must compare current `pnpm brain:inbox -- --json` paths against ledger paths:

```text
Exact match       -> resume
Mismatch          -> explain added/removed paths and ask whether to reset
```

The agent must never silently reconcile a stale ledger, discard prior checklist state, or infer that an unchecked stale line still represents the authoritative queue.

## Success criteria

v0.2 is complete when:

1. `.claude/commands/digest.md` exists and defines the `/digest` workflow.
2. `brain/projects/janus/digest-workflow.md` documents the shipped digest workflow.
3. `brain/projects/janus/sketches/v0.2.md` matches the implemented ledger/checklist design.
4. `pnpm brain:digest:init` creates `.janus/digest-ledger.md` from the same shared inbox queue function used by `pnpm brain:inbox -- --json`.
5. `brain:digest:init` preserves an existing `.janus/digest-ledger.md` by default.
6. `pnpm brain:digest:init -- --reset` overwrites an existing `.janus/digest-ledger.md`.
7. `brain:digest:init` writes the ledger atomically so generation failure does not truncate the prior ledger.
8. `.janus/digest-ledger.md` contains one checklist section with every queued inbox note as an unchecked exact path on initial creation/reset.
9. `brain:digest:init` excludes `journal/**` from the checklist.
10. `brain:digest:init` is not affected by unrelated `brain/**` or `journal/**` discovery failures.
11. `.janus/digest-ledger.md` is ignored by git through `/.janus/digest-ledger.md`.
12. `/digest` instructs the agent to use `brain:inbox -- --json` as the authoritative queue and not read `journal/**` for digestion.
13. `/digest` compares current inbox paths against ledger paths before resuming.
14. `/digest` asks the user whether to reset when ledger paths and current inbox paths differ.
15. `/digest` instructs the agent to update the checklist before moving to the next inbox note.
16. Before final approval, only `.janus/digest-ledger.md` may be modified.
17. Knowledge-source files remain unchanged until the user explicitly approves a complete changeset.
18. Post-approval workflow runs `pnpm brain:index` and reports the final diff.
19. No new retrieval/search infrastructure, background service, task runner, index schema, taxonomy, automatic tag/task system, or agent-test runner is introduced.

## Phase 2 — Shared inbox queue function

### Work

Add or refactor a shared library function equivalent to:

```ts
listInboxDocuments(root): Promise<InboxDocument[]>
```

Both `brain:inbox` and `brain:digest:init` must call that exact function.

The function must represent the digest queue directly: root inbox documents only. Do not implement digest initialization by loading broad index candidates from `brain/` and `journal/` and filtering afterward.

### Acceptance

- `brain:inbox -- --json` and `brain:digest:init` use the same queue source.
- The shared function cannot be affected by unrelated `brain/**` or `journal/**` failures.
- Queue ordering remains deterministic.
- Existing inbox JSON output remains stable.

## Phase 3 — Digest ledger initializer

### Work

Create:

```text
tools/brain/digest-init.ts
```

Update:

```text
package.json
.gitignore
```

Add package script:

```json
"brain:digest:init": "tsx tools/brain/digest-init.ts"
```

`brain:digest:init` should:

1. Resolve the repository root using existing tooling helpers.
2. Get inbox documents from the shared inbox queue function.
3. Create `.janus/` if needed.
4. If `.janus/digest-ledger.md` does not exist, write a new ledger atomically.
5. If `.janus/digest-ledger.md` exists and `--reset` is absent, preserve it and print that the digest ledger is being resumed.
6. If `.janus/digest-ledger.md` exists and `--reset` is present, overwrite it atomically.
7. Emit a concise human summary to stdout.

The generated file should use this shape:

```md
# Janus Digest Ledger

- [ ] first-note.md
- [ ] second-note.md
```

During digest execution, valid outcome lines are:

```md
- [x] first-note.md — clarified
- [x] second-note.md — parked
- [x] third-note.md — keep raw
- [x] fourth-note.md — proposed discard
```

Do not include full note bodies, retrieval evidence, user answers, or organization proposals.

Add this gitignore rule:

```gitignore
/.janus/digest-ledger.md
```

### Acceptance

- Running `pnpm brain:digest:init` creates `.janus/digest-ledger.md` when no ledger exists.
- Running `pnpm brain:digest:init` preserves an existing `.janus/digest-ledger.md`.
- Running `pnpm brain:digest:init -- --reset` overwrites an existing `.janus/digest-ledger.md`.
- The ledger includes root inbox notes from the shared queue used by `pnpm brain:inbox -- --json`.
- The ledger excludes protected root Markdown files.
- The ledger excludes `journal/**`.
- The generated checklist order is deterministic.
- Atomic write behavior preserves the prior ledger if generation fails.
- `.janus/digest-ledger.md` does not appear in `git status --short` after generation.

- `/digest` asks one targeted teacher question per turn with the harness `ask` tool.
- `/digest` enters Phase 2 for every unresolved inbox note and shows a teaching card before asking.
