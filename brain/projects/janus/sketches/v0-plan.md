# Janus v0 Implementation Plan

This plan implements the locked decisions in `brain/projects/janus/sketches/v0-qna.md` plus the review amendments for the first Root-Drop Inbox Model slice.

## Goal

Build the smallest end-to-end Janus v0 tooling surface:

- `pnpm brain:inbox`
- `pnpm brain:index`
- strict root Markdown inbox detection
- deterministic local `data/brain-index.json`
- minimal agent/wiki entrypoints
- Vitest coverage for the implemented behavior

## Non-goals

Do not implement:

- `brain:search`
- `brain:promote`
- `brain:links`
- graph tooling
- duplicate detection
- backlink/link detection
- source-code indexing
- full wiki skeleton
- `.claude/rules` files
- `.agents/skills` files
- local-only capture conventions
- full Markdown body indexing
- derived tags, embeddings, search scores, backlinks, or authority weights

## Repository-root resolution

The public v0 CLIs do not need a `--root` flag.

Root resolution contract:

1. Start from `process.cwd()`.
2. Walk upward until finding a directory containing both:
   - `package.json`
   - `AGENTS.md`
3. Treat that directory as the Janus repository root.
4. If no such directory is found, print an error to stderr and exit non-zero.

Do not resolve root using only `package.json`; nested packages may exist later.

Do not resolve root using only `AGENTS.md`; unrelated parent directories may contain one.

Tests may pass an explicit repository root into internal library functions.

## Architecture

Keep CLI entrypoints thin. Shared behavior lives under `tools/brain/lib/`.

```text
tools/brain/
├── inbox.ts              # CLI args + formatting only
├── index.ts              # CLI args + write orchestration only
└── lib/
    ├── filesystem.ts     # repo paths, file walking, root resolution, atomic writes
    ├── frontmatter.ts    # frontmatter-lite parser
    ├── classify.ts       # protected_root / inbox / wiki / archive
    ├── documents.ts      # document records, title, age, discovery, content hash
    └── output.ts         # human + JSON output helpers
```

No classification or frontmatter parsing logic should be duplicated between `inbox.ts` and `index.ts`.

## Data contracts

### Location class

Each indexed document receives exactly one `location_class`:

```ts
type LocationClass = "protected_root" | "inbox" | "wiki" | "archive";
```

Classification rules:

```text
root protected Markdown file     → protected_root
root non-protected Markdown file → inbox
brain/archive/**/*.md            → archive
all other brain/**/*.md          → wiki
```

Do not infer note `kind`, `status`, or authority from the file path beyond `location_class`.

### Brain document

The v0 index stores minimal Markdown-derived fields only:

```ts
interface BrainDocument {
  path: string;
  location_class: LocationClass;
  title: string;
  frontmatter: Record<string, string>;
  content_hash: string;
  frontmatter_warnings: string[];
}
```

Do not store in the v0 index:

- filesystem `modified_at`;
- full Markdown content;
- derived tags;
- links;
- embeddings;
- backlinks;
- search scores;
- authority weights.

`content_hash` is a SHA-256 hash of the original raw Markdown file bytes. The raw file is canonical; do not normalize parsed Markdown before hashing.

### Index file

`data/brain-index.json` has this top-level shape:

```json
{
  "schema_version": 1,
  "documents": []
}
```

The generated index must be deterministic:

```text
same Markdown files + same path structure = same index JSON
```

Rules:

- document records are sorted by normalized repository-relative POSIX `path`;
- sorting uses raw code-point ordering, not locale-aware sorting;
- do not use `localeCompare()`;
- use a comparator equivalent to `a.path < b.path ? -1 : a.path > b.path ? 1 : 0`;
- no top-level build timestamp;
- no filesystem `modified_at` field.

## CLI JSON contracts

### `brain:inbox -- --json`

Print exactly one valid JSON object to stdout. Print no human prose on stdout in JSON mode.

Shape:

```json
{
  "inbox_count": 2,
  "hygiene_status": "healthy",
  "documents": [
    {
      "path": "janus-memory-idea.md",
      "title": "Janus memory idea",
      "project": "janus",
      "age_days": 1,
      "age_source": "created",
      "frontmatter_warnings": []
    }
  ]
}
```

Types:

```ts
type InboxHygieneStatus = "healthy" | "review_soon" | "action_required";
type AgeSource = "created" | "filesystem_mtime";
```

Rules:

- records are path-sorted using raw code-point ordering;
- warnings live in structured document output;
- errors go to stderr and return non-zero;
- `project` is included when frontmatter `project` exists, otherwise omitted or `null` consistently.

### `brain:index -- --json`

Print a summary, not the full generated index:

```json
{
  "output_path": "data/brain-index.json",
  "document_count": 6,
  "counts_by_location_class": {
    "protected_root": 1,
    "inbox": 2,
    "wiki": 2,
    "archive": 1
  },
  "warning_count": 1
}
```

Rules:

- the full index remains in `data/brain-index.json`;
- no human prose on stdout in JSON mode;
- errors go to stderr and return non-zero.

## Implementation phases

### Phase 1 — Project scaffolding

1. Create `package.json`, `pnpm-lock.yaml`, `tsconfig.json`, and `vitest.config.ts`.
2. Add scripts for `brain:inbox`, `brain:index`, and `test`.
3. Add Node and pnpm pinning.
4. Create `data/.gitignore`:

```gitignore
*
!.gitignore
```

Acceptance:

- `data/` generated content is ignored.
- Root `*.md` files are not ignored.
- `pnpm` scripts exist.
- `packageManager`, `engines.node`, and `@types/node` are configured.

### Phase 2 — Classification and path rules

Implement `tools/brain/lib/classify.ts` and supporting filesystem helpers.

Protected root basenames are exact and case-sensitive:

```text
AGENTS.md
CLAUDE.md
README.md
CONTRIBUTING.md
LICENSE.md
CHANGELOG.md
CODE_OF_CONDUCT.md
SECURITY.md
```

Filesystem constraints:

- only regular files are eligible;
- directories are ignored;
- symlinks are ignored;
- matching is case-sensitive and requires exact `.md` suffix;
- hidden Markdown files like `.scratch.md` count as inbox notes unless explicitly protected;
- stored paths use repository-relative POSIX separators.

Boundary cases:

```text
README.MD              → not an inbox item
notes.md.bak           → not an inbox item
.scratch.md            → inbox item
brain/archive.md       → wiki, not archive
brain/archive/a.md     → archive
brainx/example.md      → not indexed
docs/example.md        → not indexed
AGENTS.md              → protected_root
agents.md              → inbox
```

Acceptance:

- each indexed document receives exactly one `location_class`;
- no source files or non-`brain/` nested Markdown files are indexed;
- path ordering is deterministic and locale-independent.

### Phase 3 — Frontmatter-lite parser

Implement `tools/brain/lib/frontmatter.ts`.

Rules:

- frontmatter is recognized only when line 1 is exactly `---`;
- a UTF-8 BOM before `---` is not supported in v0 and means no frontmatter is recognized;
- the block ends at the next line exactly equal to `---`;
- entries are flat `key: value` pairs;
- split on the first `:` only;
- `key: value: with colon` parses as key `key`, value `value: with colon`;
- keys must match `[A-Za-z][A-Za-z0-9_-]*`;
- empty values are valid strings equal to `""`;
- values are strings only;
- no arrays, nesting, multiline values, anchors, or type coercion;
- duplicate keys produce warnings;
- duplicate-key policy: last valid occurrence wins;
- malformed frontmatter produces warnings but does not prevent indexing.

Closed block recovery:

```md
---
project: janus
not valid metadata
status: active
---
```

Behavior:

- record a warning for the malformed line;
- ignore that malformed line;
- continue parsing later valid key-value lines.

Unclosed block recovery:

```md
---
project: janus

# My actual note
```

Behavior:

- record one `unclosed frontmatter` warning;
- treat the entire file as Markdown body;
- do not trust or extract any metadata from the malformed block;
- permit H1 title extraction from the entire original file.

Acceptance:

- malformed frontmatter does not crash `brain:inbox` or `brain:index`;
- warnings are attached to the relevant document;
- title extraction still works for unclosed frontmatter files.

### Phase 4 — Document model, title, age, and hash

Implement `tools/brain/lib/documents.ts`.

Title resolution order:

1. frontmatter `title`;
2. first H1 heading;
3. filename converted into a readable fallback title.

`created` support:

- support only date-only ISO form `YYYY-MM-DD`;
- validate structurally and semantically;
- interpret valid values as midnight UTC;
- use `created` only for inbox age calculation;
- invalid values produce a warning and fall back to filesystem modification time.

Age output for inbox:

```ts
interface InboxDocument {
  path: string;
  title: string;
  project?: string;
  age_days: number;
  age_source: "created" | "filesystem_mtime";
  frontmatter_warnings: string[];
}
```

Rules:

- `age_days` is the floor of elapsed whole 24-hour periods;
- calculate against an injectable `now`;
- negative values clamp to `0`;
- human output may render `today`, `1 day`, or `4 days`;
- avoid unstable relative phrases like `yesterday` or `3 hours ago`.

Hashing:

- compute `content_hash` using SHA-256 over raw file bytes;
- include `content_hash` in `data/brain-index.json`;
- do not include `content_hash` in human inbox output unless needed for debugging.

Acceptance:

- documents are sorted lexicographically by normalized path with raw code-point ordering;
- invalid optional `created` values become warnings and fall back to filesystem modification time;
- unchanged Markdown content produces stable content hashes.

### Phase 5 — `brain:inbox`

Implement `tools/brain/inbox.ts`.

Default human output includes:

- relative path;
- title;
- age;
- optional `project`;
- root hygiene count and status;
- parse warning, when applicable.

Support:

```text
pnpm brain:inbox -- --json
```

Root hygiene enum and display:

```text
0–10 root inbox notes: healthy          → healthy
11–19 root inbox notes: review soon     → review_soon
20+ root inbox notes: action required   → action_required
```

Acceptance:

- warnings do not fail the command;
- unreadable eligible Markdown files fail the command;
- repository/tooling failures exit non-zero;
- JSON output matches the specified contract.

### Phase 6 — `brain:index`

Implement `tools/brain/index.ts`.

Output path:

```text
data/brain-index.json
```

Write behavior:

1. create `data/` when missing;
2. create a unique temporary file inside `data/`;
3. write the complete JSON content;
4. close the file;
5. rename it to `data/brain-index.json`;
6. delete the temporary file on failure when possible.

The temporary file must be in the same directory as the final output so rename stays on the same filesystem.

If index generation fails before rename, preserve any prior valid `data/brain-index.json`.

Do not claim crash-proof durability under power loss. The v0 goal is avoiding a truncated final index during ordinary process interruption or write failure.

Support:

```text
pnpm brain:index -- --json
```

Acceptance:

- the index is deterministic for unchanged Markdown files and path structure;
- the final output file is valid JSON on success;
- write or serialization failures exit non-zero;
- unreadable eligible Markdown files fail non-zero and do not replace an existing index.

### Phase 7 — Minimal knowledge entrypoints

Create root `AGENTS.md` with the Janus rules from the v0 sketch:

- `brain/` contains promoted durable wiki knowledge;
- non-default root Markdown files are inbox notes;
- root inbox notes may be incomplete, incorrect, or unverified;
- source code and repository-local documentation remain authoritative for code behavior;
- do not create `brain/inbox/`.

Create minimal `brain/HOME.md`:

- identifies Janus as a Markdown-first personal engineering knowledge system;
- links only to existing sketch files;
- states that root Markdown files are capture inbox notes and `brain/` is durable memory.

Acceptance:

- `AGENTS.md` is classified as `protected_root`;
- `brain/HOME.md` is classified as `wiki`;
- `brain/HOME.md` contains no broken required links.

Add exact classification boundary tests:

```text

notes.md.bak           → not an inbox item
.scratch.md            → inbox item
```

Add targeted filesystem integration tests for:

## CLI failure contract

Warnings do not fail commands:

- malformed frontmatter;
- duplicate frontmatter keys;
- invalid optional `created` values;
- missing title fallback conditions.

Errors fail commands with a non-zero exit code:

- repository root cannot be resolved;
- required paths cannot be read;
- an eligible Markdown file is discovered but cannot be read;
- `data/brain-index.json` cannot be written;
- JSON serialization fails;
- unexpected filesystem errors.

If an eligible Markdown file is discovered but cannot be read, the command fails non-zero. `brain:index` must not replace an existing index in that case.

Malformed contents are warnings. Unreadable source files are errors.

## Verification checklist

Run and observe:

```text
pnpm install
pnpm test
pnpm brain:inbox
pnpm brain:inbox -- --json
pnpm brain:index
pnpm brain:index -- --json
```

Then verify:

- `data/brain-index.json` exists after `brain:index`.
- `data/brain-index.json` has `schema_version: 1`.
- documents are sorted by `path` using raw code-point ordering.
- no top-level build timestamp exists.
- no filesystem `modified_at` field exists in document records.
- every document has `content_hash`.
- JSON CLI modes print no human prose on stdout.
- root inbox notes remain visible in `git status`.
- generated `data/brain-index.json` is ignored by Git.

## Done definition

The v0 slice is done when:

1. `brain:inbox` discovers only regular, direct-root files ending exactly in `.md`, excluding exact protected basenames.
2. `brain:index` discovers only protected root Markdown, root inbox Markdown, and Markdown under `brain/`.
3. Every discovered document receives exactly one stable `location_class`.
4. Malformed frontmatter creates document warnings without preventing discovery, title resolution, inbox listing, or index generation.
5. An unreadable eligible Markdown file causes an explicit failure and preserves any existing valid index.
6. JSON interfaces are structured, path-sorted, and free of human prose on stdout.
7. `data/brain-index.json` is written atomically from deterministic Markdown-derived fields.
8. `AGENTS.md` and `brain/HOME.md` exist, are correctly classified, and contain no broken required links.
9. `pnpm test`, `pnpm brain:inbox`, `pnpm brain:inbox -- --json`, `pnpm brain:index`, and `pnpm brain:index -- --json` all pass.
10. `data/brain-index.json` is ignored by Git, while root-drop inbox notes remain ordinary tracked Markdown files.
